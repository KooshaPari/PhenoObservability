# ADR-001: Code-First Grafana Dashboard Generation

## Status
Accepted

## Context
The Phenotype platform requires observability dashboards that:
1. Are version-controlled alongside application code
2. Can be deployed automatically via CI/CD
3. Maintain consistency across environments
4. Support environment-specific configuration
5. Enable rapid iteration on visualizations

We evaluated approaches:
- Manual GUI creation with JSON export
- Terraform Grafana provider
- Jsonnet/Grafonnet
- Go code generation (current approach)
- Direct API management

## Decision
Use Go code to generate Grafana dashboard JSON, treating dashboards as code artifacts that are version-controlled, reviewed, and deployed through CI/CD pipelines.

## Rationale

### Why Code-First Over GUI

1. **Version Control**: Dashboard changes are tracked in Git with full history
2. **Code Review**: Changes reviewed through PR process
3. **Consistency**: Same dashboards across dev/staging/production
4. **Reusability**: Template patterns for common visualizations
5. **Automation**: CI/CD integration for deployment

### Why Go Over Alternatives

| Approach | Pros | Cons | Decision |
|----------|------|------|----------|
| GUI Export | Zero learning curve | Manual process, drift | Rejected |
| Terraform | Infrastructure integration | HCL limitations | Partial |
| Jsonnet | Powerful abstraction | Additional toolchain | Rejected |
| Go | Native language, type-safe, testing | More verbose | Accepted |

### Architecture Decisions

1. **Native Go Types**: Define dashboard structure as Go structs
   - Type safety
   - IDE support
   - Easy testing

2. **Functional Configuration**: Builder pattern for dashboard construction
   - Fluent API
   - Default values
   - Composable

3. **JSON Generation**: Convert Go structures to Grafana JSON
   - Standard library encoding/json
   - Validation before output
   - Pretty printing for debugging

## Consequences

### Positive
- Dashboards as first-class code artifacts
- Type-safe dashboard construction
- Unit testing for dashboard logic
- Native Go tooling (formatting, linting)
- No additional dependencies

### Negative
- More verbose than specialized DSLs
- Manual work to support new Grafana features
- Learning curve for team members
- Initial development time

### Mitigations
- Clear documentation and examples
- Library of reusable panel templates
- Code generation for repetitive patterns
- Regular updates to support new features

## Implementation Notes

### Core Types

```go
// Dashboard represents a Grafana dashboard
type Dashboard struct {
    UID         string              `json:"uid"`
    Title       string              `json:"title"`
    Tags        []string            `json:"tags"`
    Time        TimeSettings        `json:"time"`
    Panels      []Panel             `json:"panels"`
    SchemaVersion int               `json:"schemaVersion"`
}

// Panel represents a dashboard panel
type Panel struct {
    ID          int                 `json:"id"`
    Type        string              `json:"type"`
    Title       string              `json:"title"`
    GridPos     GridPos             `json:"gridPos"`
    Targets     []Target            `json:"targets"`
}
```

### Builder Pattern

```go
dashboard := NewDashboard("phenotype-operational").
    WithTag("operational").
    WithPanel(NewGraphPanel("Request Rate").
        WithTarget(PrometheusTarget("rate(http_requests_total[1m])")).
        AtPosition(0, 0, 12, 8),
    )
```
