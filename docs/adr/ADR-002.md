# ADR-002: PhenoObservability Alert Management

## Status

**Accepted**

## Context

Alert fatigue is a common problem in observability. Poorly configured alerts lead to noise, ignored notifications, and delayed response to real issues. A systematic approach to alert management is essential.

Key challenges:

1. **Alert Fatigue**: Too many alerts reduce responsiveness
2. **False Positives**: Incorrectly configured alerts waste time
3. **Routing**: Getting alerts to right team quickly
4. **Context**: Alerts without context hard to act on
5. **Runbooks**: Standardized response procedures
6. **Metrics**: Measuring alerting effectiveness

## Decision

We will implement a structured alert management system:

### 1. Alert Classification

**Rationale**: Not all alerts are equal.

**Decision Details**:
- **Critical**: User-facing outage, immediate response required
- **Warning**: Degraded service, respond within 1 hour
- **Info**: Anomaly detected, investigate within 24 hours
- Page vs notify distinction
- Business hours vs 24/7 routing
- Alert priority based on service criticality

### 2. Alert Configuration Standards

**Rationale**: Standardized configuration reduces errors.

**Decision Details**:
- All alerts defined as code (Terraform/Pulumi)
- Required fields: summary, description, runbook_link
- Alert grouping by service/cluster
- Anti-flapping with hysteresis
- Auto-resolution when condition clears
- Required review for new critical alerts

### 3. Routing and Escalation

**Rationale**: Right person at right time.

**Decision Details**:
- Primary on-call rotation per service
- Escalation to secondary after 15 minutes
- Manager escalation after 30 minutes
- Service ownership from service catalog
- Timezone-aware scheduling
- Override capability for planned maintenance

### 4. Context Enrichment

**Rationale**: Alerts must be actionable.

**Decision Details**:
- Link to relevant dashboards
- Recent deployment correlation
- Similar past incidents
- Affected user count estimate
- Related error logs excerpt
- Suggested first investigation steps

### 5. Alert Analysis

**Rationale**: Continuous improvement of alerting.

**Decision Details**:
- MTTR (Mean Time To Recovery) tracking
- Alert frequency by type
- False positive rate measurement
- Alert coverage analysis
- Regular alert review meetings
- Quarterly alert cleanup

## Consequences

### Positive

1. **Focus**: Important alerts get attention
2. **Speed**: Faster incident response
3. **Quality**: Fewer false positives over time
4. **Knowledge**: Runbooks capture expertise
5. **Burnout**: Reduced alert fatigue for on-call
6. **Coverage**: Systematic alert review ensures gaps filled

### Negative

1. **Maintenance**: Requires ongoing attention
2. **Complexity**: Sophisticated routing rules
3. **Configuration**: Many alerts to maintain
4. **Disagreement**: Team debates over alert severity
5. **Coverage Gaps**: Missed scenarios possible

### Neutral

1. **Tools**: PagerDuty/OpsGenie required
2. **Rotation**: Fair on-call scheduling needed
3. **Compensation**: On-call compensation policy

## Alternatives

### Alternative 1: Email Only

**Considered**: All alerts via email.

**Rejected Because**:
- Easy to miss
- No escalation
- Poor context
- Hard to track

### Alternative 2: Page Everything

**Considered**: PagerDuty for all alerts.

**Rejected Because**:
- Severe alert fatigue
- High cost
- Teams ignore pages

### Alternative 3: No On-Call

**Considered**: No formal on-call rotation.

**Rejected Because**:
- Uneven burden distribution
- No guaranteed response
- Knowledge silos

### Alternative 4: Automated Only

**Considered**: Only auto-remediation, no human alerts.

**Rejected Because**:
- Complex issues need humans
- False confidence in automation
- Edge cases not covered
