# ADR-003: PhenoObservability Distributed Tracing Strategy

## Status

**Accepted**

## Context

Distributed tracing is essential for understanding request flows through microservices. A well-designed tracing strategy provides visibility without overwhelming storage or impacting performance.

Key challenges:

1. **Sampling**: Full tracing too expensive at scale
2. **Instrumentation**: Consistent tracing across services
3. **Storage**: Trace data volume can be massive
4. **Correlation**: Linking traces to logs and metrics
5. **Head-Based vs Tail-Based**: When to sample decisions
6. **Performance**: Tracing overhead on services

## Decision

We will implement a tiered tracing strategy:

### 1. Sampling Strategy

**Rationale**: Balanced visibility and cost.

**Decision Details**:
- **Head-based**: 1% default sampling for all requests
- **Error-based**: 100% sampling for error responses
- **User-based**: 100% for specific user cohorts (opt-in debug)
- **Adaptive**: Dynamic rate based on throughput
- **Custom**: Per-service override capability
- Always sample for critical paths (payment, auth)

### 2. Instrumentation Standards

**Rationale**: Consistent data enables correlation.

**Decision Details**:
- OpenTelemetry as instrumentation standard
- Automatic instrumentation for HTTP/gRPC
- Manual instrumentation for business logic spans
- Required attributes: service.name, span.kind, error flag
- Optional: user.id, tenant.id, request.id
- Baggage propagation for cross-cutting data

### 3. Trace Storage

**Rationale**: Efficient storage for query patterns.

**Decision Details**:
- Jaeger/Tempo for trace storage
- 7-day hot storage for recent traces
- 30-day warm storage for investigation
- 1-year cold archive in object storage
- Trace-derived metrics for long-term trends
- Batch writes to reduce overhead

### 4. Correlation Architecture

**Rationale**: Traces alone are insufficient.

**Decision Details**:
- Trace ID in all logs for correlation
- Span ID for precise log-to-trace linking
- Trace-derived metrics (latency histograms)
- LogQL queries using trace IDs
- Grafana integration for trace-to-metrics
- Service graph generation from traces

### 5. Performance Budget

**Rationale**: Tracing must not impact service SLAs.

**Decision Details**:
- Target: <1% latency overhead
- Target: <5% CPU overhead
- Async span reporting
- UDP transport for agent communication
- Circuit breaker if collector unavailable
- Resource quotas for tracing agents

## Consequences

### Positive

1. **Visibility**: Request flow understanding
2. **Debugging**: Root cause in distributed systems
3. **Performance**: Latency analysis by component
4. **Architecture**: Service dependency discovery
5. **Optimization**: Bottleneck identification
6. **Cost**: Sampling controls costs

### Negative

1. **Complexity**: Instrumentation overhead
2. **Sampling**: May miss rare issues
3. **Storage**: Even sampled, storage significant
4. **Performance**: Some overhead unavoidable
5. **Expertise**: Requires distributed systems knowledge

### Neutral

1. **Tools**: Jaeger/Tempo/Zipkin choice
2. **Sampling**: Configurable per environment
3. **Retention**: Business-driven retention periods

## Alternatives

### Alternative 1: No Sampling

**Considered**: Sample 100% of all requests.

**Rejected Because**:
- Prohibitive cost at scale
- Storage explosion
- Query performance degradation

### Alternative 2: Application-Only

**Considered**: No infrastructure tracing.

**Rejected Because**:
- Misses database/network latency
- Incomplete request picture
- Hard to identify infrastructure issues

### Alternative 3: Metrics Only

**Considered**: Only metrics, no distributed tracing.

**Rejected Because**:
- Cannot trace individual requests
- Hard to debug distributed issues
- No request flow visibility

### Alternative 4: Log-Based Tracing

**Considered**: Derive traces from structured logs.

**Rejected Because**:
- Higher overhead than native tracing
- Complex log processing
- Less precise timing
