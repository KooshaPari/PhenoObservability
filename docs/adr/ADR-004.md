# ADR-004: PhenoObservability SLO and Error Budgets

## Status

**Accepted**

## Context

Service Level Objectives (SLOs) and error budgets provide a data-driven approach to reliability. They balance feature velocity against reliability investment and guide decision-making about risk.

Key challenges:

1. **Definition**: Defining meaningful SLOs
2. **Measurement**: Accurate SLO calculation
3. **Governance**: Enforcing error budget policies
4. **Communication**: Explaining SLOs to stakeholders
5. **Trade-offs**: Reliability vs velocity decisions
6. **Evolution**: Updating SLOs as service matures

## Decision

We will implement SLO-based reliability engineering:

### 1. SLO Definition Framework

**Rationale**: Structured approach to SLO creation.

**Decision Details**:
- SLI (Service Level Indicator): Measurable metric
- SLO (Service Level Objective): Target percentage
- SLA (Service Level Agreement): External commitment
- Start with availability and latency SLOs
- User-centric SLOs over infrastructure metrics
- SLOs reviewed quarterly
- Error budget: 1 - SLO (e.g., 0.1% for 99.9% SLO)

### 2. SLO Calculation

**Rationale**: Consistent calculation across services.

**Decision Details**:
- Availability: successful requests / total requests
- Latency: percentage under threshold (p95, p99)
- Rolling 30-day window for error budgets
- Burn rate calculation for alert sensitivity
- Multi-window, multi-burn-rate alerts
- SLO dashboards for all services

### 3. Error Budget Policy

**Rationale**: Automated enforcement prevents erosion.

**Decision Details**:
- Error budget exhausted: halt non-critical deploys
- 2% budget remaining: require VP approval for deploys
- 5% budget remaining: increased scrutiny, war rooms
- Budget reset monthly or quarterly
- Exception process for security patches
- Budget reports to leadership monthly

### 4. Reliability Tiers

**Rationale**: Not all services need same reliability.

**Decision Details**:
- **Critical**: 99.99% (52 minutes downtime/month)
- **Standard**: 99.9% (43 hours downtime/month)
- **Development**: 99% (7 hours downtime/month)
- Tier determines error budget policy strictness
- Tier determines on-call requirements
- Tier review during service registration

### 5. SLO Communication

**Rationale**: SLOs affect business decisions.

**Decision Details**:
- SLOs published in service catalog
- Error budget status in deployment tooling
- Executive dashboard with key SLOs
- SLO impact statements for launches
- Post-incident SLO impact analysis
- Training on SLO concepts for teams

## Consequences

### Positive

1. **Alignment**: Shared reliability understanding
2. **Automation**: Error budget drives decisions
3. **Focus**: SLOs focus on user experience
4. **Balance**: Data-driven reliability vs velocity
5. **Prediction**: Early warning via burn rate
6. **Communication**: Clear reliability language

### Negative

1. **Complexity**: SLO implementation overhead
2. **Debate**: SLO negotiations consume time
3. **Gaming**: Teams may optimize for SLO not user
4. **Blame**: Error budgets can feel punitive
5. **Rigidity**: Hard to change established SLOs

### Neutral

1. **Tools**: Requires SLO tooling (Sloth, Pyrra)
2. **Training**: Education required for all teams
3. **Exceptions**: Emergency process needed

## Alternatives

### Alternative 1: Uptime Only

**Considered**: Only track uptime percentage.

**Rejected Because**:
- Doesn't capture partial degradation
- Ignores performance
- Not user-centric

### Alternative 2: No Error Budgets

**Considered**: Track SLOs but no budget policy.

**Rejected Because**:
- No enforcement mechanism
- Gradual reliability erosion
- Launch/velocity decisions arbitrary

### Alternative 3: Infrastructure Metrics

**Considered**: SLOs based on CPU, memory, disk.

**Rejected Because**:
- Not user-visible
- May not correlate with user experience
- False sense of reliability

### Alternative 4: Manual Governance

**Considered**: Leadership decides on launch risk.

**Rejected Because**:
- Inconsistent decisions
- No data-driven approach
- Slow decision process
