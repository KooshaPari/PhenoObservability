# ADR-001: PhenoObservability Platform Architecture

## Status

**Accepted**

## Context

The Phenotype ecosystem requires unified observability across distributed services, applications, and infrastructure. A centralized observability platform must collect, store, and analyze telemetry data to enable troubleshooting, performance optimization, and operational intelligence.

Key challenges:

1. **Data Volume**: High cardinality metrics and high volume logs
2. **Correlation**: Linking traces, metrics, and logs across services
3. **Retention**: Long-term storage cost vs query performance
4. **Multi-Tenancy**: Isolated observability for different teams
5. **Real-Time**: Near real-time alerting on critical issues
6. **Cost**: Observability infrastructure can be expensive

## Decision

We will implement a modern observability platform:

### 1. Data Collection Layer

**Rationale**: OpenTelemetry standardizes collection.

**Decision Details**:
- OpenTelemetry agents for all services
- Auto-instrumentation for common frameworks
- Prometheus exposition for metrics
- Fluent Bit for log collection
- Custom exporters for legacy systems
- Agent deployment via DaemonSet/operators

### 2. Storage Architecture

**Rationale**: Specialized storage per telemetry type.

**Decision Details**:
- **Metrics**: VictoriaMetrics/Prometheus for TSDB
- **Logs**: ClickHouse/Loki for log aggregation
- **Traces**: Jaeger/Tempo for distributed tracing
- **Profiles**: Parca/Pyroscope for continuous profiling
- Hot/warm/cold tiering for cost optimization
- Object storage (S3) for long-term archive

### 3. Query and Analysis

**Rationale**: Unified query interface simplifies usage.

**Decision Details**:
- Grafana as primary visualization platform
- Tempo for trace search and analysis
- LogQL/PromQL for log and metric querying
- Correlation between signals (trace to logs/metrics)
- Dashboard as code (Grafana Terraform provider)
- Alerting rules as code

### 4. Alerting and Incident Response

**Rationale**: Automated detection and response.

**Decision Details**:
- Prometheus Alertmanager for metric alerts
- Grafana OnCall/PagerDuty integration
- Alert routing based on service ownership
- Runbook links in alerts
- Auto-remediation for known issues
- Incident timeline from correlated signals

### 5. Cost Optimization

**Rationale**: Observability costs grow with scale.

**Decision Details**:
- Metric cardinality limits per service
- Log sampling for high-volume services
- Aggregation for long-term metrics
- Compression (Zstd) for storage
- Down-sampling for historical queries
- Usage dashboards for chargeback

## Consequences

### Positive

1. **Visibility**: Complete system observability
2. **Debugging**: Faster root cause analysis
3. **Proactive**: Alerting before user impact
4. **Standards**: Open standards prevent lock-in
5. **Scale**: Handles high volume efficiently
6. **Cost**: Optimized storage reduces costs

### Negative

1. **Complexity**: Multiple components to manage
2. **Infrastructure**: Requires dedicated resources
3. **Overhead**: Telemetry collection adds overhead
4. **Expertise**: Requires observability expertise
5. **Cost**: Still significant at scale

### Neutral

1. **Vendor**: Can choose managed or self-hosted
2. **Retention**: Tradeoffs between cost and history
3. **Sampling**: Affects statistical accuracy

## Alternatives

### Alternative 1: Vendor SaaS

**Considered**: Use Datadog, New Relic, Dynatrace.

**Rejected Because**:
- Vendor lock-in
- Cost at scale
- Data privacy concerns
- Less customization

### Alternative 2: ELK Stack Only

**Considered**: Elasticsearch, Logstash, Kibana.

**Rejected Because**:
- Poor metrics support
- High resource usage
- Licensing changes
- Not cloud-native

### Alternative 3: No Centralization

**Considered**: Each service manages own observability.

**Rejected Because**:
- Cannot correlate across services
- Inconsistent tooling
- Hard to troubleshoot distributed issues

### Alternative 4: Commercial Only

**Considered**: Only use open source, no paid options.

**Rejected Because**:
- Support burden
- Feature limitations
- Maintenance overhead
