# ADR-001: Alert Rule DSL Design

## Status: Accepted

**Date:** 2026-04-05  
**Author:** Phenotype Engineering Team  
**Project:** alerting

---

## Context

The alerting library needs a domain-specific language (DSL) for defining alert rules that can express complex conditions while remaining readable and maintainable.

### Requirements

1. **Prometheus Compatible**: Rules should work with Prometheus/Alertmanager
2. **Type Safe**: Compile-time validation of expressions
3. **Composable**: Build complex rules from simple components
4. **Readable**: Rules should be self-documenting

### Constraints

- Go 1.21+ for generics support
- YAML output for Prometheus compatibility
- Must support standard observability patterns

---

## Decision

We will use a **Go struct-based DSL with YAML serialization**:

```
┌────────────────────────────────────────────────────────────────┐
│              Alert Rule DSL Architecture                        │
│                                                                 │
│  Go Structs ─────▶ YAML ─────▶ Prometheus                     │
│                                                                 │
│  ┌─────────────┐   ┌─────────────┐   ┌─────────────┐          │
│  │   Alert     │   │  YAML       │   │ Alertmanager│          │
│  │   Struct    │──▶│  Format     │──▶│  Config     │          │
│  │             │   │             │   │             │          │
│  │ - Name      │   │ groups:     │   │ groups:     │          │
│  │ - Expr      │   │ - name:     │   │ - name:     │          │
│  │ - Labels    │   │   rules:    │   │   rules:    │          │
│  │ - Annotations│  │   - alert:  │   │   - alert:  │          │
│  └─────────────┘   │     expr:   │   │     expr:   │          │
│                    │     ...     │   │     ...     │          │
│                    └─────────────┘   └─────────────┘          │
└────────────────────────────────────────────────────────────────┘
```

### Rationale

| Option | Pros | Cons | Verdict |
|--------|------|------|---------|
| **Go DSL** | Type-safe, IDE support | Go knowledge required | **SELECTED** |
| Pure YAML | Standard, simple | No validation, errors at runtime | Rejected |
| PromQL | Powerful, standard | Complex learning curve | Partial |
| CEL | Flexible, fast | Extra dependency | Future |

### Implementation

```go
type Alert struct {
    Name        string            `yaml:"name"`
    Labels      map[string]string `yaml:"labels"`
    Annotations map[string]string `yaml:"annotations"`
    Expr        string            `yaml:"expr"`
    For         string            `yaml:"for"`
}

type AlertRuleSet struct {
    Groups []AlertGroup `yaml:"groups"`
}

// Builder pattern for fluent construction
func HighErrorRate(threshold float64) Alert {
    return Alert{
        Name: "HighErrorRate",
        Expr: fmt.Sprintf("rate(errors[5m]) > %f", threshold),
        For:  "5m",
        Labels: map[string]string{
            "severity": "critical",
        },
    }
}
```

---

## Consequences

### Positive

- **Type safety**: Compile-time validation
- **IDE support**: Autocompletion, navigation
- **Testability**: Rules can be unit tested
- **Composability**: Functions create reusable patterns
- **Documentation**: Code comments explain rules

### Negative

- **Go required**: Users must know Go
- **Compilation step**: Rules not dynamic
- **Verbosity**: More verbose than YAML

### Mitigations

1. Provide rule templates/examples
2. Runtime rule validation
3. Optional YAML loading for dynamic rules
